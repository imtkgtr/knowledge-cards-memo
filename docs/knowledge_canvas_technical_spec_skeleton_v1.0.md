# 知識キャンバスアプリ 技術仕様書 骨組み v1.0

## 1. 文書の目的
本書は、基盤仕様書で定義したプロダクト要件を、実装可能な技術仕様へ落とし込むための**骨組み**である。ここでは、実装に必要な責務、機能領域、技術的に決めるべき項目、受け入れ条件、未確定事項の置き場を整理する。

本書は、採用技術やライブラリを確定する文書ではない。具体的な実装手段や技術選定は、別途 Codex 等で調査し、本書の各章に追記・確定していく前提とする。

---

## 2. 技術仕様書の最重要目的
技術仕様書として最も重要なのは、**基盤仕様で定めた体験を壊さずに、実装責務を明確に分割し、迷わず開発できる状態を作ること**である。

このアプリにおける実装上の最重要軸は次の通りである。

1. 中央キャンバスが主役であること
2. ページではなくキャンバスを操作する体験であること
3. カード・リンク・配置・可視化が一貫したデータとして扱えること
4. 保存・復元・Undo/Redo が破綻しないこと
5. クイック操作と詳細編集の責務が混ざらないこと

## 4. 技術仕様として守るべき前提

### 4.1 ワークスペース前提
- キャンバス画面はビューポート固定
- body 全体のスクロールは発生させない
- 移動はキャンバスのパン・ズームで行う
- スクロール可能なのは、検索候補、右パネル、モーダル内部など局所領域のみ

### 4.2 データ一貫性前提
- カード、リンク、背景、色、ロック、配置は、すべて 1 キャンバス単位で保存・復元できること
- JSON エクスポート / インポートでは添付ファイルを扱わない
- Undo/Redo と自動保存が矛盾しないこと

### 4.3 編集責務分離前提
- 左上パレットはクイック操作
- 右パネルは詳細編集
- 検索は移動用であり、編集開始操作ではない
- 複数選択時は右パネルを閉じ、専用操作バーへ責務を移す

## 5. システム全体構成（骨組み）

### 5.1 想定コンポーネント
本アプリは少なくとも次の論理コンポーネントで構成する。

1. 認証層
2. キャンバス一覧管理層
3. キャンバス編集層
4. 保存・復元層
5. 添付ファイル管理層
6. JSON 入出力層
7. Undo/Redo 管理層
8. 描画 / 表示制御層

### 5.2 画面単位の責務
- ログイン画面: 認証のみ
- キャンバス一覧画面: キャンバスの作成・選択・管理
- キャンバス画面: 編集・閲覧・配置・検索・整列・書き出し

### 5.3 状態の大分類
実装時には、状態を少なくとも次の3種に分けて扱う。

- 永続化される状態
- 画面セッション中だけの状態
- 一時的な UI 状態

## 6. 主要機能一覧

- ログイン
- ログアウト
- 継続ログイン
- 任意の外部ログイン追加余地

技術仕様で決めること:

- セッション管理
- エラーハンドリング
- 保護ルート設計

### 6.2 キャンバス管理機能
必要な機能:
- キャンバス作成
- キャンバス一覧表示
- キャンバス選択
- 名前変更
- 削除
- 複製
- サムネイル表示
- インポート / エクスポート

技術仕様で決めること:
- 一覧取得方法
- サムネイル生成と更新タイミング
- 複製処理方式
- 削除確認と論理削除 / 物理削除

### 6.3 キャンバス編集機能

- 無限キャンバス表示
- パン
- ズーム
- ミニマップ
- 背景設定
- カード配置
- カード移動
- クイックパレット
- 右パネル編集

技術仕様で決めること:
- 描画ライブラリ or DOM / Canvas / SVG 方針
- レンダリング最適化
- 座標系設計

### 6.4 カード管理機能
必要な機能:
- カード作成
- タイトル、本文、タグ編集
- 色変更
- ロック / 解除

技術仕様で決めること:

- カードサイズ算出方式
- タイトル折り返し処理

必要な機能:
- 階層リンク追加
- 通常リンク追加

- 同種重複リンク禁止
- 循環禁止（階層リンク）

### 6.6 タグ機能

- タグ強調
- タグ絞り込み

技術仕様で決めること:

- 強調 / 非表示の具体描画

### 6.7 複数選択機能

## 8. データモデル（骨組み）
本節は、厳密な型定義ではなく、技術仕様書で型へ落とすためのモデル分解である。

## 8.1 Canvas
保持したい情報:
- id
- name
- createdAt
- updatedAt
- thumbnail
- background
- gridEnabled
- metadata（必要なら）

## 8.2 Card
保持したい情報:
- id
- canvasId
- title
- body
- tags
- color
- locked
- x
- y
- createdAt
- updatedAt
- attachmentSummary（必要なら UI 用）

## 8.3 HierarchyLink
保持したい情報:
- id
- canvasId
- parentCardId
- childCardId
- createdAt

## 8.4 RelatedLink
保持したい情報:
- id
- canvasId
- cardAId
- cardBId
- createdAt

## 8.5 Attachment
保持したい情報:
- id
- cardId
- kind
- displayName
- fileName
- mimeType
- size
- storageRef
- createdAt

注記: JSON には含めない。

## 8.6 ExportPayload
保持したい情報:
- version
- canvas
- cards
- hierarchyLinks
- relatedLinks

注記: 添付は含めない。

## 9. 永続化される状態 / されない状態

## 9.1 永続化される状態
- キャンバス情報
- カード情報
- カード座標
- リンク
- 色
- ロック状態
- 背景設定
- タグ
- 本文

## 9.2 永続化しない状態
- 右パネル開閉中か
- 左パレット開閉中か
- リンク追加モード中か
- 検索入力途中状態
- 一時的な強調表示状態（仕様次第）

## 9.3 要検討状態
- アコーディオン開閉状態を保持するか
- ミニマップの表示 / 非表示を保持するか
- 同名警告の「今後表示しない」の保存場所

## 11. 保存仕様の骨組み

### 11.1 自動保存
決めるべきこと:
- 何操作ごとに保存するか
- 何秒停止で保存するか
- パン / ズームを保存対象にするか
- 保存中表示の要否

### 11.2 手動保存
決めるべきこと:
- 明示保存ボタンの配置
- 保存成功 / 失敗表示

### 11.3 競合回避
決めるべきこと:
- 複数タブで開いた場合の扱い
- 最終更新競合の扱い

---

## 12. JSON 入出力仕様の骨組み

### 12.1 エクスポート
必要条件:
- 1 キャンバス単位
- `version` 付き
- 添付なし
- 背景 / 色 / ロック / 座標 / リンクを含む

### 12.2 インポート
必要条件:
- 新規キャンバスとして追加
- バリデーション失敗時は作成しない
- エラーメッセージ表示

### 12.3 技術仕様で決めること
- JSON Schema
- バージョン差異の扱い
- 不正IDの再採番有無

---

## 13. 非機能要件

### 13.1 可用性
- 通常操作でデータ消失しにくいこと
- 自動保存で基本的に内容が保持されること

### 13.2 性能
- カード追加、移動、色変更、ロック、検索が体感的に重すぎないこと
- 大量ノード時も最低限編集可能であること

### 13.3 一貫性
- 左はクイック操作、右は詳細編集という責務が崩れないこと
- 検索が勝手に編集を始めないこと

### 13.4 安全性
- 未対応添付は拒否すること
- 不正 JSON は弾くこと
- ロックされたカードは守られること

### 13.5 保守性
- キャンバス、カード、リンク、保存、Undo/Redo の責務が分離されていること

## 14. テスト観点

### 14.1 画面遷移
- ログイン成功後に一覧へ行ける
- 一覧からキャンバスを開ける
- 戻る導線が機能する

### 14.2 カード
- タイトル未入力では作れない
- 同名警告が出る
- 作成位置が現在表示範囲中心になる
- ロックカードは編集不可

### 14.3 リンク
- 自己リンク禁止
- 重複リンク禁止
- 階層循環禁止

### 14.4 タグ
- 強調と絞り込みが意図どおり分かれる
- AND 条件が正しく効く

### 14.5 保存
- 自動保存が動く
- 手動保存できる
- JSON が出力できる
- JSON から新規キャンバスとして復元できる

### 14.6 Undo/Redo
- 主要操作が戻せる
- やり直せる

### 14.7 表示
- ミニマップが同期する
- パン / ズームが機能する
- ページ全体スクロールが発生しない
